# Expansion 03 Implementation Plan: Райсовет и Живой архив

Статус: implementation planning document  
Scope: playable MVP документов, доступов и архива внутри существующего `MINISTRY`  
Запрет: новый `FloorLevel.ADMIN` не вводится до отдельного доказанного room/pocket slice.

## Техническая цель

MVP должен дать игроку бюрократический путь решения задач, который работает не как текстовая шутка, а как механическая альтернатива бою, торговле и обходу. Минимальная проверяемая петля: игрок получает легальный пропуск, получает или делает рискованный поддельный документ, применяет оба к разным проверкам доступа, видит последствия в suspicion, реакции охраны, журнале событий и debug-командах.

Реализация должна быть data-driven и дешевой по умолчанию. Документ проверяется в момент взаимодействия с дверью, NPC, контейнером или квестовым узлом. Архив не сканирует мир каждый кадр; он возвращает bounded карточки из известных NPC, quest facts, важных world events и локальных архивных записей. Очереди и бюро не симулируют сотни граждан: они представляются несколькими NPC, короткими состояниями комнаты и редкими событиями.

## Релевантные мандаты

Для этого slice применяются шесть мандатов из пользовательского протокола и локальных design-документов. `MINISTRY-only` не дает распухнуть scope и запрещает преждевременный `ADMIN`. `Bounded state` ограничивает архив, события и debug-буферы. `Zero-GC runtime` требует предсозданных document defs, массивов и lookup id вместо временных объектов на hot path. `Math LOD` задает четыре уровня логики и визуала. `Debug visibility` требует команд на выдачу, проверку и порчу документов. `Black Box` фиксируется как контракт для будущей реализации: последние 300 проверок документов и архивных запросов должны быть дампируемыми, но в текущей задаче создается только документационная спецификация.

## Phase 0: Contract Freeze

Перед кодом нужно зафиксировать маленький интерфейс документов, доступа и архива, не завязанный на несуществующие модули. Основные данные: `DocumentPermitDef`, `DocumentInstance`, `AccessCheckRequest`, `AccessCheckResult`, `ArchiveQuery`, `ArchiveCard`. Источник истины по определениям документов должен жить в data-файле, runtime-инстансы должны ссылаться на `defId`, а не копировать длинные строки.

DOD этой фазы: в `integration_contract.md` описаны структуры данных, правила failure mode, правила интеграции с `MINISTRY`, запрет `ADMIN`, debug hooks и минимальный черный ящик. Отвергнутая альтернатива: завести документ как произвольную строку в инвентаре, потому что такой объект нельзя надежно проверять у двери, портить самосбором или показывать в debug.

Проверка: статическая ревизия contract-файла против `README.md`, `desdoc.md`, root system docs, `Docs/Expansions/INDEX.md` и текущего `Docs/Expansions/03_raionsovet_archive/expansion.md`.

## Phase 1: Paper MVP Data

Первый кодовый срез должен начинаться не с UI, а с восьми document defs и компактной таблицы access tags. Обязательный набор: временный пропуск в архив, пропуск в бюро, санитарная справка, акт реквизиции, предписание ликвидатора, архивная карточка NPC, фальшивый пропуск, испорченная форма. Каждый документ обязан иметь хотя бы один механический эффект: открыть доступ, снизить штраф, повысить suspicion, разрешить контейнер, изменить цену, создать событие или разблокировать диалог.

MVP допускает хранение документов как item/note hybrids, если отдельный inventory type еще не готов. В этом случае `item.id` или metadata содержит `documentDefId`, а все проверки идут через document-system helper. Текст формы остается коротким: название, issuer, access tags, suspicion, срок, inspection text.

DOD этой фазы: минимум 8 document defs, минимум 8 access tags, нет проверки по display title, debug-команда может выдать любой document def игроку. Отвергнутая альтернатива: 100 документов сразу. Это расходует production time на flavor до появления playable loop.

Проверка: unit-style ручной список `debug.giveDocument(defId)`, `debug.inspectDocument(defId)`, `debug.listDocuments()`; все id стабильны и не конфликтуют с expansion ids других агентов.

## Phase 2: Access Checks

Access check должен быть единой функцией, которую можно вызвать из двери, NPC-охранника, картотеки, контейнера и квестового узла. Request содержит context: actor, target kind, floor, room id, zone id, required tags, faction, current hour, samosbor variant. Result возвращает allow/deny, reason code, suspicion delta, consumed document flag, event id и короткий HUD text.

Логика low-tier: поиск документа у игрока по tag и сроку. Middle добавляет suspicion, подделку и штрафы. High добавляет faction/reputation modifiers и последние важные события. Ultra добавляет визуальные проверки в комнате, но не меняет сложность hot path.

DOD этой фазы: одна дверь или контейнер в `MINISTRY` открывается легальным пропуском, другая проверка принимает фальшивку с риском, охранник реагирует на отказ без постоянного polling. Отвергнутая альтернатива: копировать проверки в каждый генератор комнаты, потому что это создаст несовместимые правила доступа.

Проверка: debug-команда `checkAccess tag target` возвращает reason code; ручные сценарии показывают `allow`, `deny_missing_tag`, `deny_expired`, `allow_forged_suspicious`.

## Phase 3: Ministry Rooms Slice

Контент MVP расширяет существующий `MINISTRY`: райсоветская очередь, бюро пропусков, комната печатей, живой архив и один проверяющий пост. Эти комнаты должны быть маленькими, подключенными к текущему ministry generator и не требовать нового этажа. Если генератор занят другим агентом, slice может стартовать как независимый module spec с graceful no-op registration.

Игровая последовательность: очередь дает задачу на простой пропуск; бюро выдает легальный документ после короткого fetch/talk; комната печатей дает рискованный обход через фальшивку; архив возвращает карточку NPC или квестового факта; проверяющий пост демонстрирует последствия. Бумага должна хотя бы один раз быть лучше боя: доступ без агрессии и без расхода патронов. Бумага должна хотя бы один раз создать проблему: фальшивка повышает suspicion, портит отношение или вызывает событие проверки.

DOD этой фазы: 4 места, 4 named NPC, 1 guard checker, 1 legal path, 1 forged path, 1 archive lookup. Отвергнутая альтернатива: большой этаж Райсовета, потому что индекс expansion требует MVP без нового большого этажа.

Проверка: path smoke test от входа в очередь до проверяющего поста; debug warp или room marker допустим для разработки.

## Phase 4: Living Archive

Архив хранит агрегаты, а не полную историю. MVP-источники: known named NPC, завершенные quest ids, последние important world events при наличии `world_log`, локальные штрафы, выданные пропуска и статические архивные записи. Query возвращает одну короткую `ArchiveCard`, где последняя зона агрегирована, опасность может быть слуховой, а доступ представлен tag-ами.

Архив может ошибаться после самосбора. Это не random noise каждый запрос; ошибка фиксируется флагом `warped`, `stale`, `duplicate` или `future_dated`, чтобы debug мог объяснить результат. Запросы должны иметь cooldown или interaction-only trigger.

DOD этой фазы: запрос по одному NPC и одному квестовому факту; карточка показывает имя, последнюю известную зону, долг/штраф, риск, связанные access tags; самосбор может испортить минимум два типа записи. Отвергнутая альтернатива: база всех координат NPC, потому что она ломает mystery и добавляет лишний runtime cost.

Проверка: `debug.archiveQuery npc:<id>`, `debug.archiveQuery quest:<id>`, `debug.warpArchive <mode>`, сравнение карточки до и после мутации.

## Phase 5: Samosbor And Consequences

Самосбор должен портить бюрократию механически. Минимум два эффекта для MVP: мокрый или классический вариант портит печати и делает часть документов недействительной; тихий или электрический вариант создает ложный приказ, который может пропустить игрока один раз, но повышает suspicion или вызывает охрану позднее.

Эффекты не должны пересчитывать весь инвентарь каждый кадр. Мутация документов выполняется при окончании самосбора, при входе в affected ministry room или при первой проверке документа после события. Document instance получает флаги `warped`, `wet`, `stampDamaged`, `falseOrder`, `expiresAtHour`.

DOD этой фазы: два samosbor mutation rules, один HUD message, один world/archive event, debug-команда принудительной порчи. Отвергнутая альтернатива: физическая симуляция бумаги, воды и печатей. Дешевый флаг дает тот же gameplay сигнал.

Проверка: форсировать вариант самосбора, выдать документ, проверить доступ до и после mutation, убедиться в понятном reason code.

## Phase 6: Debug, Telemetry, And Black Box

Debug должен быть полноценной частью MVP. Нужны команды выдачи документа, списка документов игрока, проверки access tag, порчи печати, запроса архива и вывода последних решений access system. Для будущей реализации критическая система должна иметь fixed-size circular telemetry на 300 записей: time, actor id, target id, def id, required tag hash, result code, suspicion delta, flags. На NaN или impossible state дамп пишется в `Docs/AgentLogs/Dump_EXP03_RAIONSOVET.bin`.

В TypeScript/Vite MVP можно начать с bounded array и numeric enum codes; позднее заменить на typed ring buffer без изменения контракта. Главное: не оставлять access denial необъяснимым.

DOD этой фазы: debug-команды видны, последние решения можно вывести, reason codes документированы, dump contract описан. Отвергнутая альтернатива: полагаться на console.log, потому что отчеты других агентов и CTO смотрят файлы и debug tools, а не эфемерный terminal output.

Проверка: ручной прогон всех debug-команд и проверка отсутствия allocation-heavy строк на hot path в будущей кодовой ревизии.

## Math LOD

Low-tier режим делает документы статичными item/note hybrids, проверяет access только при interaction, хранит архивные карточки как заранее собранные записи и показывает очередь без социальной симуляции. Цель для слабых устройств: до 0.02 ms на одиночную проверку и нулевой per-frame cost вне UI.

Middle-tier добавляет suspicion score, expiry, несколько штрафов, реакцию охраны и две samosbor mutation rules. Очередь имеет 2-4 NPC с hand-written репликами, но не симулирует толпу.

High-tier подключает последние important world events, NPC-memory facts для доносов, фракционные modifiers и влияние документов на цены или квестовые ветки. Архив остается lookup-based и обновляется только по событию.

Ultra-tier покупает атмосферу сохраненными циклами: лампы ЭВМ, перфоленты, мерцающие таблицы, цифровые призраки DATA и визуальная деградация печатей. Логика не становится глобальным поиском по миру; overkill остается визуальным и локальным.

## Risks

Первый риск: бумажный UI станет рутиной. Контрмера: документы короткие, проверяются контекстно, в HUD попадает только reason text и consequence.

Второй риск: бюрократия останется flavor. Контрмера: каждый document def имеет mechanical effect и test scenario.

Третий риск: раннее добавление `ADMIN` расколет архитектуру этажей и работу других агентов. Контрмера: все MVP-модули пишутся как расширения `MINISTRY`, а поздний `ADMIN` допускается только после отдельного contract review.

Четвертый риск: архив превратится в дорогую omniscient database. Контрмера: bounded facts, cooldowns, static records, агрегированные зоны вместо точных координат.

## Verification Matrix

MVP считается playable, когда проверены пять сценариев. Легальный путь: игрок получает пропуск в бюро и проходит guard/access point без агрессии. Поддельный путь: игрок получает фальшивку, проходит или проваливает проверку с suspicion consequence. Архивный путь: игрок запрашивает NPC или quest fact и получает короткую карточку. Самосборный путь: документ портится или мутирует, result code меняется. Debug path: команда может выдать документ, проверить tag, испортить печать и показать последние access decisions.

Build verification для документационного PR: `npm run build` должен проходить или, если упал из-за чужих кодовых изменений, статус фиксирует external compile wall без правки чужого кода.
