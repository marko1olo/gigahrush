## Текущая задача
**Система: Объединение ручного крафта и фабричного производства в ЕДИНУЮ систему CraftRecipe.**



### Контекст задачи
Сейчас ручной крафт (`src/systems/crafting.ts` + `src/data/craft_recipes.ts`) и фабричное производство (`src/systems/production.ts` + `src/data/factories.ts`) — это ДВЕ разные системы с разными интерфейсами, разными data-форматами и разной логикой. Нужно ОБЪЕДИНИТЬ их в ОДНУ систему с единым `CraftRecipeDef`, единым registry и единым runtime.

### Конкретные файлы и паттерны

#### Шаг 1: Единый рецепт
- **`src/data/craft_recipes.ts`**: Расширить `CraftRecipeDef` полем `stationType: CraftStationKind | 'hand' | 'factory'`. Рецепты с `'hand'` — ручной крафт (без станка). Рецепты с конкретной станцией — требуют стоять рядом. Рецепты с `'factory'` — требуют production room.
- **`src/data/factories.ts`**: Мигрировать 42 фабричных рецепта в единый `CRAFT_RECIPES` массив с `stationType` = соответствующий factory type.

#### Шаг 2: Единая runtime логика
- **`src/systems/crafting.ts`**: Расширить `craftItem()` — при наличии `stationType !== 'hand'` проверять, что игрок стоит рядом с нужной станцией (проверка через `world.features` или production room proximity).
- **`src/systems/production.ts`**: Оставить как thin wrapper для NPC-автоматической фабрики (автономные production rooms). Но РЕЦЕПТЫ берутся из единого `CRAFT_RECIPES`.

#### Шаг 3: Единый UI
- **`src/render/craft_ui.ts`**: Показать ВСЕ известные рецепты. Рецепты, требующие станцию рядом, greyed out если станции нет. Фильтры: «Все» / «Доступные сейчас» / по категории.

#### Ключевое
- **Не менять save shape**: `CraftingState` хранит `knownRecipes: string[]` — это уже работает. Фабричные рецепты просто добавляются в тот же массив.
- **Backward compatible**: Старые recipe id-шники не меняются.
- **Тесты**: `npm run typecheck` + `npm run test:unit` обязательно.
