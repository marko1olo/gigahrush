# Аномалии процедурных этажей

Плановый документ. Код не менять в рамках этого файла.

Процедурный этаж строится обычным способом: `ProceduralFloorSpec` выбирает геометрию, большинство фракции, опасность, лут и монстров; `generateProceduralFloor()` создает 1024x1024 тороидальный `World`, комнаты, коридоры, зоны, лифты и начальные сущности. Аномалия должна накладываться поверх уже готовой геометрии как отдельный профиль `FloorAnomalyId`, без отдельного `FloorLevel` и без сюжетной комнаты.

Общий контракт для всех субпроектов:

- Данные: новый id в `src/data/procedural_floors.ts`, вес, `minDanger`, `dangerBias`, теги лута/монстров/слухов.
- Генерация: маленькая `apply...()` функция в `src/gen/procedural_floor.ts` или отдельный модуль, вызываемый оттуда. Лифты, стартовая зона и обязательные проходы защищаются.
- Runtime: один generic runtime-канал в `src/systems/procedural_anomalies.ts` или узкий модуль, который он вызывает. Никаких `setInterval`, DOM UI и полных сканов мира каждый кадр.
- Хранение: плотные карты только если они действительно нужны на клетку; иначе списки индексов, `Map<number, ...>`, ring buffer, seed и компактное состояние.
- Тороид: любые координаты через `world.idx`, `world.wrap`, `world.delta`, `world.dist`, `world.dist2`.
- Рендер: использовать существующие `Cell`, `Tex`, `Feature`, `fog`, `light`, `surfaceMap`, marks, HUD/log. Новые атласы только если эффект нельзя выразить текущими средствами.
- Игровое решение: каждая аномалия должна дать выбор: обойти, использовать, починить, наживить врагов, спрятаться, украсть, сесть, выключить, переждать или сбежать.
- Бюджет: генерация может быть тяжелее, runtime должен работать на медленных тиках, dirty-списках, чанках, радиусах и лимитах.

## 1. `conway_life` - Игра жизнь

Идея: обычный бетонный этаж заражен клеточным автоматом. После генерации часть комнат и коридоров получает слой "живого бетона"; в реальном времени по нему идут правила Conway Game of Life: живая клетка выживает при 2-3 соседях, мертвая рождается при 3 соседях. Живая клетка проявляется как временная стеновая масса, мертвая как проход.

Данные профиля:

- HUD title: `игра жизнь`.
- Рекомендуемые теги: `cellular`, `topology`, `moving_walls`, `math`.
- `minDanger`: 3.
- `dangerBias`: 2.
- Лучше редкая аномалия, потому что она меняет проходимость и требует runtime.

Генерация:

- Обычная геометрия строится первой.
- Выбирается 1-4 арены: большие комнаты, цепочки коридоров или квадратные блоки 32x32/64x64.
- Для каждой арены создается `candidateMask`: только клетки, где безопасно менять `FLOOR <-> WALL`.
- Из маски исключить лифты, стартовый радиус, двери, hermo walls, важные контейнеры, клетки под сущностями на старте.
- Начальное состояние лучше брать не напрямую из всех стен, а из процедурного seed-шаблона: шум, границы комнат, редкие "организмы", несколько глайдеров.
- На полу оставить метки-сетки и один аппарат "глушитель автомата".

Runtime:

- Тик 0.5-1.0 секунды, не каждый кадр.
- Не считать весь 1024x1024 мир. Для каждой арены держать два `Uint8Array` буфера ее локального размера и dirty border.
- При тике считать только арену и менять только клетки, состояние которых изменилось.
- Запретить рождение стены в радиусе 2 клеток от игрока, лифта и открытой двери.
- Если клетка стала стеной под NPC/монстром, не телепортировать: выталкивать в ближайшую floor-клетку радиусом 3 или наносить дробящий урон, но игрока не убивать мгновенно.
- Сущности могут использовать аномалию: монстры запираются, NPC паникуют, игрок может наживить погоню в будущую стену.

Решения игрока:

- Пройти по ритму, когда коридор открыт.
- Выключить арену аппаратом, потратив инструмент или редкий предмет.
- Намеренно кормить автомат: бросать предметы/трупы как "живые" seed-клетки, чтобы открыть проход или закрыть погоню.
- Рисковать коротким путем через активный паттерн.

Рендер и обратная связь:

- Живые клетки: темный бетон/мясной налет, щелкающие marks, локальный шум HUD.
- Перед тиком мигать клетками, которые изменятся, чтобы смерть не была скрытой.
- В debug summary показывать число арен, активных клеток, dirty-клеток и следующий тик.

Тесты и риски:

- Unit: детерминированный B3/S23 на маленькой локальной арене.
- Generator: лифты и spawn не попадают в mutable mask.
- Smoke/check: визуально проверить, что карта не становится пустой или полностью запертой.
- Главный риск: ломает навигацию. Поэтому это "аренная" аномалия, а не автомат на весь этаж.

## 2. `wall_snake` - Змейка

Идея: по этажу ездит змейка из бетонных блоков. Голова управляется компьютером, тело занимает клетки как движущаяся стена. Змейка ест предметы, NPC, монстров и игрока; после еды растет. Ее можно использовать как дробилку и транспортную угрозу.

Данные профиля:

- HUD title: `змейка`.
- Теги: `moving_walls`, `predator`, `crush`, `loot_sink`.
- `minDanger`: 2.
- `dangerBias`: 2.

Генерация:

- Выбрать один связный контур или сеть широких коридоров. Узкие комнаты не подходят.
- Проставить "яблоки": ценный лут, приманки, NPC/монстры в зоне змейки.
- Создать стартовое тело 8-24 клетки, голова не ближе 48 клеток к spawn.
- Тело на старте не перекрывает лифты, двери и единственный проход из стартовой комнаты.
- Добавить 1-2 безопасные ниши, куда змейка не заходит.

Runtime:

- Тик 0.25-0.8 секунды по опасности этажа.
- Тело хранить как кольцевой буфер cell indices. На каждом шаге меняются только новая голова и старый хвост.
- Клетки тела временно становятся стеной или отдельным blocking-feature; если менять `world.cells`, обязательно bump нужных версий/света точечно.
- AI головы выбирает цель из bounded списка ближайших съедобных сущностей/предметов, без поиска по всему миру каждый тик.
- Путь: простая жадная навигация по тороидальной дистанции плюс запрет самоедства; для сложных мест короткий BFS в радиусе, не глобальный pathfind.
- Еда: item drop исчезает; NPC/монстр получает огромный урон; игрок при контакте получает предупреждение/урон, а не тихую мгновенную смерть на первом касании.
- Рост ограничен максимумом, например 96/160/256 сегментов по danger.

Решения игрока:

- Бросить предмет как приманку.
- Завести змейку на монстра или вооруженный патруль.
- Перебежать через хвост, когда он освобождает клетку.
- Попробовать убить голову ради редкого лута, но тело продолжает двигаться несколько тиков.

Рендер и обратная связь:

- Голова: `Feature.SCREEN`/аппарат с красным глазом или процедурный sprite.
- Тело: темный бетон с номерными marks, на миникарте короткая пульсирующая линия.
- HUD prompt рядом с приманкой: "бросить еду".

Тесты и риски:

- Unit: ring buffer роста/сжатия, запрет перекрытия лифтов.
- Runtime: змейка не аллоцирует массивы на каждом шаге.
- Риск: запирание игрока. Нужны safe niches и правило "не ставить голову в клетку игрока без предыдущего предупреждения".

## 3. `bad_apple_world` - Bad Apple!

Идея: этаж становится физическим экраном. Кадры Bad Apple! последовательно печатаются в мире ГИГАХРУЩА: черные/белые силуэты превращаются в стены, пол, туман, свет или marks. Игрок идет внутри клипа, а не смотрит на DOM-видео.

Данные профиля:

- HUD title: `bad apple!`.
- Теги: `video`, `screen`, `topology`, `cult_media`.
- `minDanger`: 3.
- `dangerBias`: 1.

Источник кадров:

- Runtime не должен тянуть сеть и не должен добавлять видеодекодер.
- Источник кадров должен быть подготовлен заранее из локально предоставленного материала или легального frame pack.
- В репозиторий стоит коммитить только компактное производное представление: низкое разрешение, RLE/delta frames, без аудио и без исходного видео.
- Если юридически нельзя хранить готовые кадры, субпроект остается dev-only importer: пользователь сам кладет источник, скрипт генерирует локальный artifact.

Генерация:

- Выбрать сцену 192x144, 256x192 или 320x240 клеток внутри 1024x1024 мира.
- Масштаб 1 клетка = 1 пиксель для "ирл" эффекта или 2x2/3x3 клетки для читаемости.
- Сцена встраивается в один крупный зал или в длинную галерею.
- Spawn и лифты остаются вне активного прямоугольника.
- Вокруг сцены нужны обходные коридоры и наблюдательные окна.

Runtime:

- 8-12 FPS максимум; на слабом устройстве можно падать до 4 FPS.
- Хранить delta frames: список изменившихся cell indices/локальных пикселей, не перерисовывать прямоугольник целиком.
- Режимы:
  - безопасный: кадры меняют `floorTex`, `fog`, `light`, marks;
  - опасный: черные пиксели становятся стенами, белые полом, но только внутри protected arena.
- Сущности в меняющихся клетках получают толчок или урон, если их зажимает силуэт.
- На последних кадрах можно усилить самосбор-таймер или спавнить экранников.

Решения игрока:

- Идти по белому кадру как по открывающейся дороге.
- Ждать нужный силуэт, чтобы открыть короткий путь.
- Запереть врагов черной формой следующего кадра.
- Выключить проектор и получить доступ к backstage-луту.

Рендер и обратная связь:

- На стенах и полу эффект должен выглядеть как клип, но оставаться читаемым в raycaster.
- HUD показывает "кадр N" только debug-режимом; игроку лучше давать diegetic projector hum и мерцание.
- Можно добавить процедурный звук щелчков без внешнего аудио.

Тесты и риски:

- Тест загрузки frame pack: детерминированный размер, frame count, delta count.
- Smoke: проверить, что canvas не черный/пустой и FPS не рушится.
- Риск: объем данных. Нужен лимит artifact, например до нескольких сотен KB с delta/RLE.

## 4. `rail_trains` - Поезда

Идея: по карте ездят поезда. Они давят NPC и монстров, могут задавить игрока, но в них можно сесть и кататься между станциями/узлами. Это не новый этаж метро, а аномалия любого процедурного этажа: рельсы прорезают существующую геометрию.

Статус реализации: `rail_trains` добавлен в `FloorAnomalyId` и процедурный deck. Генератор прорезает 1-3 фиксированных линии, ставит платформы с табло, регистрирует маршруты и спавнит составы через общий runtime `systems/rail_trains.ts`. Та же система используется на `dark_metro`: темная пересадка получает живые составы поверх авторских линий. Поезда останавливаются на станциях, дают посадку/выход по `E`, везут игрока как пассажира и наносят crush damage живым сущностям на пути.

Данные профиля:

- HUD title: `поезда`.
- Теги: `rail`, `transit`, `crush`, `industrial`.
- `minDanger`: 2.
- `dangerBias`: 1.
- Чаще подходит `collectors`, `workshops`, `admin_pockets`; реже жилым этажам.

Генерация:

- После обычной геометрии выбрать 1-3 линии: петля, восьмерка, две станции с разворотом.
- Рельсы прокладывать по широким коридорам и через пробитые технические тоннели.
- Track хранить как упорядоченный список cell indices, а не как поиск по карте.
- Станции: platform cells, `Feature.SCREEN` расписания, контейнеры с ремонтным лутом.
- Минимум одна станция должна быть достижима от spawn без перехода через активные рельсы.

Runtime:

- Поезд хранит `trackId`, `headOffset`, `speed`, `length`, `doorState`, `passengerIds`.
- Движение поезда обновляет только сегменты головы/хвоста, аналогично змейке, но по фиксированному track.
- Collision: NPC/монстр получает crush damage; игрок сначала слышит/видит предупреждение, затем получает тяжелый урон.
- Boarding: `E` на платформе или у открытой двери медленного поезда. Пока игрок внутри, его позиция привязана к вагону и сходит на следующей платформе.
- AI может бояться рельсов или не успевать уйти в зависимости от фракции/паники.

Решения игрока:

- Ехать быстро через опасный этаж.
- Завести монстра на пути.
- Заблокировать стрелку предметом или ремонтом, чтобы изменить маршрут.
- Ограбить станционный ящик и получить конфликт с фракцией.

Рендер и обратная связь:

- Рельсы: floorTex/marks и металлические полосы.
- Поезд: несколько blocking segments, шум, вибрация HUD, свет фар.
- Миникарта показывает ближайший поезд и платформы.

Тесты и риски:

- Unit: поезд не покидает track, корректно зацикливается на тороиде.
- Runtime: boarding/unboarding не выбрасывает игрока в стену.
- Риск: слишком много cell mutation. Лучше рисовать поезд сущностями/feature overlay, а `world.cells` менять только если нужно для collision.

## 5. `fractal_floor` - Фрактал

Идея: в обычный этаж врастает самоподобная структура. Комнаты, стены, туман, лут и монстры повторяются на разных масштабах: маленькая кладовая похожа на весь этаж, коридор повторяет свою форму через 64 клетки, а центральная пустота строится как ковер Серпинского или другой простой integer-фрактал.

Данные профиля:

- HUD title: `фрактал`.
- Теги: `fractal`, `maze`, `topology`, `documents`.
- `minDanger`: 3.
- `dangerBias`: 1.

Генерация:

- Не заменять весь этаж. Выбрать fractal domain: квадрат 128/256/384 клеток или несколько вложенных комнат.
- Использовать integer-функции, которые дешевы и детерминированы:
  - Sierpinski carpet для стен/пустот;
  - dragon curve для marks и коридорного маршрута;
  - recursive room stamps для повторяющихся кладовых.
- Обязательная connected spine: один гарантированный проход через fractal domain.
- В повторяющихся копиях менять лут: первая копия настоящая, остальные "копии" с испорченными или опасными предметами.

Runtime:

- Базовый вариант почти весь generation-time.
- Опциональный runtime: "итерация" раз в 20-40 секунд открывает следующий масштаб, но меняет только заранее подготовленные cell lists.
- Навигационная аномалия: двери могут вести в уменьшенную/увеличенную копию комнаты через `anomalyTeleports`, но pairs должны быть редкими и понятными.

Решения игрока:

- Разгадать повтор и найти настоящую кладовую.
- Использовать самоподобный короткий путь.
- Взять копию предмета с риском, что она рассыпется/позовет монстра.
- Закрыть итерацию аппаратом и стабилизировать один маршрут.

Рендер и обратная связь:

- Повторяющиеся marks на полу и стенах, одинаковые комнаты с чуть разными текстурами.
- На карте фрактал должен читаться силуэтом.
- Не использовать огромные декоративные эффекты: сила аномалии в геометрии.

Тесты и риски:

- Generator: connected from spawn to both lift directions.
- Unit: integer fractal mask deterministic for seed.
- Риск: фрактал может стать скучной картинкой. Нужны игровые копии: лут, телепорты, угрозы, аппарат стабилизации.

## 6. `section_shift` - Секционный сдвиг

Идея: отдельные полосы этажа сдвигаются как панели старого лифта. Горизонтальный или вертикальный сектор шириной 4-12 клеток раз в несколько секунд уезжает на 1-3 клетки по тороиду, смещая стены, пол, двери, предметы и живых.

Данные профиля:

- HUD title: `секционный сдвиг`.
- Теги: `topology`, `moving_rooms`, `crush`, `toroid`.
- `minDanger`: 3.
- `dangerBias`: 2.

Генерация:

- Выбрать 3-8 shift bands. Полосы не пересекают лифты и стартовую комнату.
- Полосы отмечаются предупреждающими marks и световыми швами.
- В каждой полосе должен быть хотя бы один альтернативный обход рядом.
- На границах полос можно поставить контейнеры/двери, которые открываются только при удачном выравнивании.

Runtime:

- Тик 4-12 секунд; движение можно делать мгновенным с предупреждением или анимировать визуально, но логика меняется дискретно.
- Не сдвигать весь мир. Хранить список клеток полосы и выполнять bounded copy через scratch buffer.
- При сдвиге обновлять только затронутые cells/roomMap/features/fog/tex; контейнеры и сущности внутри полосы сдвигать на тот же delta.
- Если на границе возникает crush, сущность получает урон и выталкивается в ближайшую свободную клетку.
- Двери, которые оторвались от комнаты, санитизировать локально, без полного `sanitizeDoors()` каждый тик.

Решения игрока:

- Ждать нужного положения полосы, чтобы открыть путь.
- Сдвигом раздавить погоню.
- Застрять между секциями ради закрытой кладовой.
- Вырубить конкретную полосу через щиток.

Рендер и обратная связь:

- За 1-2 секунды до сдвига шов мигает, идет звук реле.
- HUD/log предупреждает только при близости игрока, чтобы не спамить.
- Миникарта может показывать активную полосу тонкой линией.

Тесты и риски:

- Unit: toroidal band shift сохраняет количество клеток и не теряет контейнеры.
- Runtime: сдвиг не трогает лифты.
- Риск: copy ошибок в typed arrays. Лучше начать с одной полосы и малых width.

## 7. `mirror_run` - Зеркальная проводка

Идея: этаж имеет зеркальную ось или пару осей. Некоторые комнаты существуют парами: действие в одной отражается в другой. Предмет, взятый слева, исчезает справа; убитый монстр может породить отражение; зеркало-проход перебрасывает игрока в симметричную клетку.

Данные профиля:

- HUD title: `зеркальная проводка`.
- Теги: `mirror`, `duality`, `teleport`, `loot`.
- `minDanger`: 2.
- `dangerBias`: 1.

Генерация:

- Выбрать ось `x = k`, `y = k` или диагональную имитацию внутри bounded domain.
- Скопировать 4-12 комнат как mirror pairs с сохранением достижимости.
- Пары хранить через cell index mapping для отдельных интерактивных клеток, а не для всего мира.
- Поставить зеркальные экраны/окна, через которые видно или слышно отраженную комнату.

Runtime:

- Список paired cells: контейнеры, двери, аппараты, редкие лут-точки.
- Interaction mirroring: открыть дверь тут -> щелкнула дверь там; забрать предмет тут -> там осталась пыль/подделка.
- Mirror jump: редкие `anomalyTeleports` в симметричную клетку.
- Монстры могут получать "отражение" при смерти, но с cooldown и лимитом.
- Не пытаться синхронизировать каждую клетку пола каждый тик.

Решения игрока:

- Решить, какую сторону ограбить первой.
- Открыть проход себе, но открыть путь врагам на отраженной стороне.
- Перейти через зеркало, чтобы обойти блокаду.
- Разбить зеркало и потерять часть лута, зато остановить дубликаты.

Рендер и обратная связь:

- Холодные marks, симметричные номера комнат, одинаковые контейнеры.
- HUD prompt на зеркале: "E: пройти в отражение" или "E: разбить".
- События публиковать как local rumors с tags `mirror`, `anomaly`.

Тесты и риски:

- Unit: mirror coordinate function respects toroidal wrap.
- Generator: paired target cell walkable before creating teleport.
- Риск: игрок не понимает связь. Нужны ранние безопасные пары: лампа/дверь/пустой ящик.

## 8. `cement_memory` - Цементная память

Идея: этаж запоминает маршрут игрока. Клетки, по которым игрок прошел, через задержку темнеют, твердеют, дымят или становятся приманкой. Чем дольше игрок петляет, тем больше пол превращается в опасную память.

Данные профиля:

- HUD title: `цементная память`.
- Теги: `trail`, `pressure`, `no_backtracking`, `samosbor`.
- `minDanger`: 3.
- `dangerBias`: 1.

Генерация:

- Геометрию почти не менять.
- Поставить несколько "амнезийных" зон: комнаты, где память стирается или замедляется.
- Добавить редкий лут: мел, растворитель, старые планы этажа.
- В опасных местах оставить альтернативные loops, чтобы аномалия не была прямым тупиком.

Runtime:

- Хранить ring buffer последних посещенных клеток игрока: cell index, timestamp, stage.
- Тик 1 секунда. Обрабатывать только новые/созревшие записи буфера.
- Стадии:
  - свежий след: mark и легкий шум;
  - твердый след: замедление или боль;
  - запекшийся след: временная wall/fog/monster lure;
  - выветривание: клетка возвращается к базовому состоянию.
- Не записывать клетку каждый кадр; только при переходе игрока в новую клетку.
- NPC/монстры реагируют на след как на запах/опасность с простыми local checks.

Решения игрока:

- Планировать маршрут без обратного хода.
- Стереть след предметом.
- Проложить опасный след как ловушку для погони.
- Забежать в амнезийную комнату и сбросить давление.

Рендер и обратная связь:

- Следы на полу через `surfaceMap`/marks, затем fog/light затем твердая фактура.
- HUD не должен рисовать tutorial; достаточно логов при первой стадии и при опасном возврате.

Тесты и риски:

- Unit: ring buffer не растет бесконечно и корректно выветривает клетки.
- Runtime: нет записи на каждом frame position jitter.
- Риск: раздражает игрока. Нужны понятные counters и decay.

## 9. `conveyor_sorter` - Сортировочный конвейер

Идея: этаж превращен в сортировочный узел. Полосы пола двигают предметы, трупы, NPC, монстров и игрока по стрелкам к сортировочным ямам, кладовым, прессам и платформам. Это аномалия про поток вещей внутри 1024x1024 клеточного мира.

Данные профиля:

- HUD title: `сортировочный конвейер`.
- Теги: `conveyor`, `items`, `industrial`, `movement`.
- `minDanger`: 2.
- `dangerBias`: 1.
- Особенно хорошо сочетается с `workshops` и `collectors`.

Генерация:

- Создать 2-5 conveyor loops или линии между комнатами.
- Каждая линия хранится как список cell indices + direction per cell.
- Поставить приемники: контейнеры, прессы, мусорные ямы, станционные ящики.
- Разложить лут так, чтобы часть сама уезжала, если игрок медлит.
- Обязательные проходы не должны требовать идти против конвейера без альтернативы.

Runtime:

- Тик 0.25-0.5 секунды.
- Обрабатывать только сущности в активных conveyor cells; для этого при движении сущности можно проверять текущую клетку в `Map<number, dir>`.
- Предметы двигаются полностью; живые получают velocity push/позиционный сдвиг.
- Прессы наносят урон с предсказуемым циклом.
- Игрок может выключать линии щитками, менять стрелку одной развилки или заклинивать полосу тяжелым предметом.

Решения игрока:

- Быстро забрать лут до сортировки.
- Отправить врагов в пресс.
- Сесть на поток как быстрый путь.
- Рискнуть и починить щиток ради доступа к кладовой.

Рендер и обратная связь:

- Стрелки на полу через marks, движущийся свет, металлический звук.
- Миникарта может показывать только ближайшие активные линии.
- Пресс должен заранее мигать.

Тесты и риски:

- Unit: direction map корректно двигает item drops по loop.
- Runtime: не аллоцировать per entity per tick.
- Риск: физика толчков конфликтует с AI. Начать с item drops и игрока, затем NPC/монстры.

## 10. `radio_chess` - Радио-шахматы

Идея: пол делится на шахматные клетки, но правила объявляет радио. Каждые несколько секунд активен другой цвет или фигура: черные клетки вредят, белые безопасны; потом наоборот; затем "конь" разрешает только клетки на определенном паттерне вокруг радиомаяков. Это не головоломка в UI, а клеточный ритм движения в мире.

Данные профиля:

- HUD title: `радио-шахматы`.
- Теги: `pattern`, `radio`, `timing`, `movement`.
- `minDanger`: 2.
- `dangerBias`: 1.

Генерация:

- Наложить шахматную/модульную разметку на bounded domains: 32x32, 64x64, длинные коридоры.
- Поставить радиомаяки: `Feature.SCREEN`/`APPARATUS`, от которых идут локальные паттерны.
- Сгенерировать 1-3 безопасные "скамейки" вне board domains.
- Лут положить так, чтобы его можно было взять в опасную фазу, но не обязательно.

Runtime:

- Глобальный или локальный phase timer 3-8 секунд.
- Не сканировать доску. Опасность клетки вычислять формулой от `(x, y, phase, seed)` при входе игрока/сущности или на медленном tick для ближайших.
- NPC/монстры получают простой modifier: одни игнорируют цвет, другие избегают активных клеток, третьи наоборот ускоряются.
- Радиомаяк можно выключить, изменить phase или украсть катушку.
- Сильный урон только после предупреждения: подсветка клетки до смены phase.

Решения игрока:

- Двигаться в ритме.
- Выключить конкретный маяк и открыть безопасное окно.
- Завести врага на цвет, который сейчас "под боем".
- Украсть катушку, но потерять подсказки радио.

Рендер и обратная связь:

- Легкая клеточная подсветка `light`/`surfaceMap`, радио щелкает перед сменой.
- Логи короткие: "Радио объявило черные клетки".
- Не превращать в tutorial overlay.

Тесты и риски:

- Unit: формула опасности детерминирована и не требует stored mask.
- Runtime: phase не зависит от frame rate.
- Риск: нечитаемость в raycaster. Нужна высокая контрастность только рядом с игроком и звуковое предупреждение.

## Общая очередность внедрения

1. Сначала добавить только data ids и generation-only версии для самых безопасных профилей: `fractal_floor`, `mirror_run`, `radio_chess`.
2. Затем runtime с bounded cell lists: `conveyor_sorter`, `cement_memory`, `rail_trains`.
3. После этого тяжелые движущиеся topology-анomalies: `wall_snake`, `conway_life`, `section_shift`.
4. `bad_apple_world` делать отдельным интеграционным субпроектом после решения вопроса с источником кадров и размером artifact.

Минимальная проверка для любого реализованного профиля: `npm run typecheck` и `npm run test:unit`. Для runtime, движения, save/load, рендера или генератора проходимости: `npm run check`.
